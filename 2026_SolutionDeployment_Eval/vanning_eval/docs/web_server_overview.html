vanning-eval — Web サーバー構成 https://136.109.238.213.sslip.io/

ローカル launch.bat 起動から、チームに URL を 1 つ配るだけで使える状態へ移行。 副次目的として GCP の基礎 (VM / Firewall / systemd / TLS) を実地で学ぶ。

3. トラフィックの流れ (1 リクエスト)

  1. ブラウザが https://136.109.238.213.sslip.io/ を解決 → IP がそのまま返る
  2. VM の Caddy:443 が TLS 終端 (Let's Encrypt の証明書を自動発行・更新)
  3. Caddy が reverse_proxy localhost:8502 で内部に流す
  4. Streamlit がレンダリング、必要なら GitHub Contents API で leaderboard を読み書き
  5. Caddy 経由でレスポンス、WebSocket もデフォルトで upgrade される

4. セキュリティの絞り方

  • FW ファイアウォール: 80/443 のみ開放。8502 直叩きは 閉鎖済み (timeout になる)
  • Bind Streamlit は 127.0.0.1:8502 にバインド。外部 NIC からは見えない
  • TLS HTTP は Caddy 側で 443 へ自動 redirect
  • Secret GitHub PAT は secrets.toml (chmod 600) に置く。SSH 鍵で VM ログインを守る前提

5. 重要な設計判断

HTTPS を Phase 1 に前倒し

当初は HTTP-only MVP で出す計画だったが、Chrome 117+ の HTTPS auto-upgrade で IP 直アクセスが白画面化することが判明。HTTP-only 公開は最初から成立しないと分かり、HTTPS を最初から組み込んだ。

nginx + certbot ではなく Caddy

Caddyfile 1 ホスト = 数行で証明書取得・更新・WebSocket upgrade まで自動。学習目的でも「Web サーバーの本質」を 1 ファイルで読める方が理解が深い。

ドメインを買わずに sslip.io

Let's Encrypt は HTTP-01 検証で FQDN を要求するため、生 IP だけでは TLS を張れない。<IP>.sslip.io が IP を返してくれるおかげでドメイン購入 0 円で TLS 化。

nohup → systemd

当初 nohup python ... で起動 → VM 再起動でアプリ消失。vanning-eval.service で常駐化、journalctl -u vanning-eval でログ統一。

6. 運用 (再アクセス時のコマンド)

# サーバー状態 (IP は動的なので毎回確認)
gcloud compute instances describe vanning-eval --zone=us-west1-a \
  --format="value(status,networkInterfaces[0].accessConfigs[0].natIP)"

# Streamlit のログ
gcloud compute ssh vanning-eval --zone=us-west1-a \
  --command="sudo journalctl -u vanning-eval -f"

# Caddy のログ (証明書更新確認など)
gcloud compute ssh vanning-eval --zone=us-west1-a \
  --command="sudo journalctl -u caddy --no-pager -n 30"

# 死活
curl -sS https://136.109.238.213.sslip.io/_stcore/health

7. 既知のリスクと未完タスク

突かれやすい弱点
  • 動的外部 IP: VM 停止 → 起動で IP が変わると sslip.io URL も変わる。$1.46/月で static IP 予約可
  • 認証なし: URL を踏める誰でもリーダーボードに書ける (PAT 経由)。教員配布前提なら IP 制限 or Basic 認証が筋
  • PAT 平文: secrets.toml にベタ書き。Secret Manager 移行は Phase 4 で予定
  • Tailscale 干渉: 136.x.x.x 帯は CGNAT と近く、Tailscale ON のブラウザでトラフィックが奪われる事例あり
残タスク (proj-vanning-eval-gcp より)
  • secrets.toml → Secret Manager 移行 (任意、学習目的)
  • PAT 期限を 30〜60 日に短縮 + ローテーション運用
  • チームへ URL 共有 + 動作確認依頼

8. 学んだこと (発表で言える話)

  • 「ローカル launch.bat → URL 1 本配布」の移行は、VM + リバプロ + systemd + TLS の 4 点セットで成立する
  • HTTPS は後付けではなく 最初から。Chrome の auto-upgrade を読み違えると MVP が白画面で全滅する
  • Caddy / sslip.io の組み合わせで ドメイン 0 円・設定 5 行で TLS まで届く。学習用構成として再利用しやすい
  • e2-micro の Always Free は本気で運用に耐える (16〜32 並行ユーザー程度の社内利用なら十分)
